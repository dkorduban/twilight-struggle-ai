## About
This document describes some recurrent pain point I want to avoid in the future. Not in particular order.

## Meta-algorithm
The key idea I want to happen is that the agents do experimentation and incremental improvement of engine/search/models etc. in a smart, cost-efficient way. But it seems that the main Claude agent is tasked with many parallel tasks: suport the flywheel + fix current issues + work on plan + answer my questions, and loses track of what needs to be done. This leads to frequent and preventable issues, like (list not complete):
- training dying and the agent not immediately noticing
- models plateuing and still agent is PPO training run after run
- models regressing and agent still do training instead of using Opus to root cause
  - and not just accepting first answer, but veryfying it's indeed helping. see v23+ epic fail that took a day and 20 wasted runs to fix 
- GPU / CPU idling while there's always useful stuff to do
- tasks / priorities being forgotten

My high-level vision is that Sonnet main agent can support the running flywheel + plan implementation, but ask Opus help in case:
1. training regresses/plateaud/unexpected results/suspicious metrics
2. anything suspicious, really
3. hard debuggin needed

At the same time my real time questions shoulnd't throw agents away from what are they doing (even if touching currently active stuff -- always have a continuation plan in head and implement it if resources are free; killing is kinda cheap)

## Pipeline and infra fragility
Rn the training > eval pipeline is very adhoc. We tried .snakemake w/ relative success, but not continued for some reason. There's also an option of Dagster etc. A number of issues:
- training breaking mid-run -> continuation issues / dangling stuff
- some runs overwriting others in output files / W&B results
- stale cache files
- broken checkpoints
- hard repdoducibility
- resource contention GPU, CPU

I want to make it more robust in a number of ways:
- more principled pipeline engine
- each model is run from clean repo, or at least git diff is posted to w&b
  - need w&b ids to be unique -- no overwriting/appending to existing run; can create new one with same name but different id
- every model checkpoint id = name + gitsha ( or even better the w&b id that hash gitsha + diff) + the checkpoint file checksum. the
  - maybe something better. but basically to prevent ever one checkpoint overwriting other, or not being able to understand how was is created
- all checksums, hyperparams, etc, logged to wandb
- a relational DB with all metadata, including win counts, etc; not lost ever. with backups!

## Experiment tracking
Experiment tracking doc must be clean and completet! git sha, exact command, W&B id, human readable explanation of the experiment, experimnet hypothesis.

Currently it's easy to forget model/agent lineage. Always start from v0 after big branch (e.g. adding new scalar features or smallchoice head); keep a nice document that describes all changes

Need to be ore aggressive at doing something else at plateaus -- we collected a lot of near identical models.

Overall good idea to /askopus when stuck or after meaningful experiment;

## Performance
Agents were too shy on optimizing latency/throughput and deciding to "just wait 2 hrs / 30 minutes...". not acceptable if solution exist to cut it 5x or even 2x. maybe overestimating impl time? most implementation is very fast by agents, like 10 min where it estimates 2 hrs, or 1 hr where it estimated 2 days. this also impact architectural decitions btw.

some ideas: start candidate / confirmation games as soon as checkpoint appears! this way they will overlap w/ training and a lot of latency will be solve. actually I want this to be done.

or just do some hyperparam tuning around parralelism; tje machine has 20 vCPUs so if CPU-heavy task doesn't load 18 of them it can be pushed further (always to nice though)
