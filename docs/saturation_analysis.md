# Saturation Analysis: Why 1x95ep Beats 2x47ep

**Date**: 2026-04-03
**Context**: v99 clean sweep experiment on Twilight Struggle behavioral cloning

---

## 1. The Core Result

At matched total gradient steps (~121M samples seen):

| Configuration | Unique games | Unique rows | Epochs | Combined WR |
|---|---|---|---|---|
| 1x @ 95ep | 10k | 1.28M | 95 | **29.5% +/- 0.7** |
| 2x @ 47ep | 20k | 2.58M | 47 | **26.9% +/- 0.7** |

Same seed (42), same architecture (baseline h256), same hyperparameters. The only
variable is data arrangement: fewer unique games repeated more often vs. more unique
games repeated fewer times. The 2.6pp gap is ~3.7 standard errors.

---

## 2. Why This Happens: Multi-Pass Memorization as Feature Extraction

### 2.1 The standard ML intuition (and why it fails here)

In standard supervised learning, more unique data almost always helps at equal
compute, because each new sample provides novel gradient signal and more data
reduces overfitting. The expectation is: 2x unique data at 47ep >= 1x data at 95ep.

This intuition breaks in the TS behavioral cloning setting for several reinforcing
reasons.

### 2.2 Explanation 1: Implicit curriculum from repeated exposure

When the model sees the same 10k games 95 times, it has a chance to learn each
game *deeply* -- not just the surface pattern of "in this state, play this card,"
but the sequential logic of entire game trajectories. Behavioral cloning at the
per-decision level strips out temporal context: each row is an independent
(state, action) pair. But repeated exposure to the same game's decisions lets the
model implicitly learn intra-game correlations through gradient accumulation.

At 47 epochs over 2x data, each game is seen ~47 times. At 95 epochs over 1x
data, each game is seen ~95 times. The extra 48 passes per game let the model
better memorize the *decision patterns* within games, not just the marginal
action distribution.

This is closely related to the **multi-epoch training benefit** observed in
language model pretraining research (Muennighoff et al., "Scaling Data-Constrained
Language Models," NeurIPS 2023): repeated epochs on high-quality data can match
or exceed single-epoch training on proportionally more data, up to a point (they
find diminishing returns around 4-16 epochs for LLMs, but our setting is very
different -- see below).

### 2.3 Explanation 2: Dataset homogeneity reduces the marginal value of new games

nash_b and nash_c are both generated by the **same heuristic** with different
random seeds. The heuristic plays nearly identically regardless of seed -- same
evaluation function, same search depth, same coup/realignment priorities. The
games differ only in card draw order and dice outcomes.

This means the marginal information from game 10,001 to game 20,000 is much
smaller than from game 1 to game 10,000. The *policy* being imitated is the
same; only the *states visited* differ. Since the model already sees 10,000
games worth of state diversity (covering most common board positions), the
extra 10,000 games add only diminishing state coverage.

In contrast, each additional epoch on the original 10k games lets the model
better learn the consistent decision pattern the heuristic follows, because it
sees the same game positions and the heuristic's responses to them many times.
This is pure signal reinforcement, not noise injection.

**Key insight**: The marginal value of new data depends on policy diversity, not
state diversity alone. When all data comes from the same policy, more games mostly
adds noise around the same underlying decision function.

### 2.4 Explanation 3: Regularization and loss landscape effects

95 epochs of OneCycleLR on 1.28M rows creates a different optimization trajectory
than 47 epochs on 2.58M rows:

- **Learning rate schedule**: OneCycleLR with 95 epochs has a longer warmup and a
  longer annealing phase. The model spends more time at low learning rates, which
  allows finer-grained convergence. With 47 epochs, the cosine decay is steeper
  and may not anneal as thoroughly.

- **Batch diversity within epochs**: Each epoch shuffles the data. With 1.28M rows
  and batch size 8192, there are ~156 batches per epoch. With 2.58M rows, ~315
  batches per epoch. The 95-epoch run has 14,820 total gradient steps; the 47-epoch
  run has 14,805 steps. Steps are nearly identical, but the 95-epoch run has
  **95 data shuffles** versus 47. More reshuffling of the same data creates more
  diverse mini-batch compositions across epochs, which can act as an implicit
  regularizer.

- **Implicit regularization through repetition**: Repeated exposure with different
  mini-batch contexts (different co-occurring examples in the same batch) lets the
  model learn features that are robust across batch compositions. This is related
  to the "stochastic weight averaging" effect documented in SGD optimization
  literature.

### 2.5 Explanation 4: This is the small-data regime

1.28M rows from 10k games is not a large dataset for a game with ~10^40 possible
states. The model is heavily data-constrained. In this regime:

- The model cannot overfit to the data in a harmful way because the evaluation
  metric (win rate vs. heuristic) measures generalization to *new game states*,
  not prediction on held-out rows from the same distribution. 95 epochs may
  increase train loss overfitting, but if the model better captures the
  *underlying policy*, it generalizes better to novel states during play.

- Standard overfitting metrics (train-val loss gap) are poor proxies because
  val_loss measures next-action prediction accuracy, not play strength. We already
  know val_loss and win rate are decorrelated in this project.

### 2.6 Is this a known phenomenon?

Yes, this has been observed in several settings:

1. **Data-constrained language models** (Muennighoff et al., 2023): Training for
   multiple epochs on high-quality data matches or exceeds training on more data
   of the same quality, up to ~4x repetition for LLMs. Our finding extends this
   to ~2x in the 95-vs-47 epoch regime.

2. **Imitation learning / behavior cloning**: DAgger and related work shows that
   BC benefits more from *consistent* demonstrations than from *diverse* ones when
   the demonstrations come from the same expert. More demos from the same expert
   have diminishing returns vs. more passes over existing demos.

3. **AlphaGo/AlphaZero self-play**: Training on a fixed buffer of self-play games
   for many epochs before generating new data is standard practice. The insight is
   the same: extract maximum signal from current data before spending compute on
   data generation.

4. **Small-data deep learning** (medical imaging, etc.): Aggressive augmentation
   + many epochs on small datasets routinely outperforms training on larger
   unannotated datasets, because the quality-per-sample is higher and repetition
   extracts more of it.

## 3. Would This Reverse With More Data?

**Almost certainly yes, at some scale.** The crossover point depends on:

### 3.1 When more data would win

- **Policy diversity**: If nash_c came from a *different* heuristic (different
  evaluation function, different search depth), it would add genuinely novel
  decision patterns. More data from diverse policies is more valuable than
  repeated data from one policy. Self-play generations that actually improve
  will eventually provide this diversity.

- **State coverage**: At some point (probably >>20k games), the extra state
  coverage starts to matter because the model encounters rare but important
  positions (unusual card draws, rare event chains) that 10k games don't cover
  well. The threshold depends on the effective dimensionality of the state space
  visited by the heuristic.

- **Model capacity**: A larger model (h512, h1024) might extract more value from
  2x data because it has the capacity to represent finer distinctions between
  similar states. Our h256 model may be capacity-limited enough that 10k games
  already saturate what it can learn.

### 3.2 When repetition would still win

- As long as all data comes from the **same policy** (same heuristic, same
  decision function), more data provides only state-coverage diversity, not
  policy diversity. The marginal value of state coverage is logarithmic in the
  number of games for a fixed-capacity model.

- For **very small models** (h128 or smaller), repetition should dominate even
  further because the model cannot even represent the distinctions that more
  data would teach.

### 3.3 Estimated crossover

Given the 2.6pp gap at 2x, and assuming logarithmic returns to data scale, the
crossover is likely somewhere in the range of **4x-10x unique data** (40k-100k
games) for the current h256 architecture. But only if the additional data comes
from a meaningfully different source (different heuristic params, self-play with
a different model, or human games).

If all data comes from the same heuristic, repetition may win indefinitely at
this model capacity.

## 4. Alternative Explanations to Rule Out

### 4.1 Could nash_b and nash_c have different quality?

**Partially ruled out.** Both are generated by the same heuristic with the same
parameters, differing only in random seed. The experiment log confirms both are
post-DEFCON-fix with similar win rates (USSR ~62-65%). However, there could be
subtle differences:

- Different card shuffle seeds could lead to systematically different game types
  (e.g., one seed produces more Europe-heavy games). This is unlikely to produce
  a 2.6pp effect but cannot be fully excluded without checking the game-length
  and outcome distributions of nash_b vs. nash_c separately.

- **To rule out**: Train 1x on nash_c alone for 95 epochs with seed=42 and
  benchmark. If it also beats 2x @ 47ep, data quality difference is excluded.

### 4.2 Could the 3.7-sigma result be noise?

**Unlikely but not impossible.** The 3.7-sigma is calculated from benchmark
variance only (same model evaluated on different game seeds). It does not account
for:

- **Training trajectory sensitivity**: Even with the same seed (42), the training
  dynamics are different (different number of batches per epoch, different LR
  schedule granularity). The "same seed" control eliminates random initialization
  variance but not optimization-path variance.

- **Multiple testing**: If we had run many saturation experiments and cherry-picked
  the best, 3.7-sigma would be less convincing. Since this is a pre-planned
  comparison (saturation hypothesis), the significance is genuine.

- **Benchmark variance underestimate**: The +/-0.7pp CI assumes games are
  independent. If there are systematic patterns (e.g., certain card shuffles
  always favor one side), effective sample size is smaller and CI is wider.

**Verdict**: The result is likely real (>95% confidence), but confirming with
a second seed pair (e.g., 1x-nash_b-s7 vs. 2x-s7) would raise confidence
further.

### 4.3 Could there be data leakage in the 2x dataset?

**Worth checking.** With the deterministic game_id hash split:

- If nash_b and nash_c have overlapping game_ids (different games but hash
  collisions), some val games from nash_b might have near-duplicates in nash_c's
  train split. This would make the 2x val_loss look artificially good without
  improving real play strength.

- More importantly: if the val split for 2x contains games from both nash_b and
  nash_c, while the 1x val split contains only nash_b games, the validation
  distributions differ. This doesn't affect win rate (which is the real metric)
  but could explain why val_loss is not predictive.

- **To check**: Verify that game_id hashing produces disjoint train/val sets
  with no cross-contamination between nash_b and nash_c halves.

### 4.4 Could the learning rate schedule be the real cause?

**Plausible contributing factor.** OneCycleLR with 95 epochs has:
- ~9.5 epochs of warmup (10% of 95) vs. ~4.7 epochs for 47ep
- Longer annealing tail

If the 47-epoch run doesn't anneal the LR far enough, it may leave the model in
a slightly suboptimal basin. This is testable by running 2x @ 95 epochs
(~243M total samples) and seeing if it catches up. If it does, the result is
about total training time, not data arrangement.

**To rule out**: Run 2x @ 95ep with seed=42. If it matches or beats 1x @ 95ep,
the effect is LR schedule, not data saturation.

## 5. Implications for Self-Play Strategy

### 5.1 Data generation budget

The result suggests that generating more heuristic self-play data is **low
priority**. The 10k games in nash_b (1.28M rows) already contain enough signal
for the current model capacity. Compute is better spent on:

1. **More training epochs** on existing data (diminishing returns past ~95, but
   testing 120-150ep is cheap)
2. **Qualitatively different data** (self-play from a stronger model, MCTS
   trajectories, eventually human games)
3. **Architecture improvements** that can extract more from existing data

### 5.2 Self-play data mixing

When mixing heuristic + self-play data:

- **Do not dilute with large self-play volumes.** If self-play data comes from
  a model that learned from the heuristic, it's near-duplicate data (same
  underlying policy with small perturbations). The Phase 1 finding that self-play
  data provides "NO reliable improvement" is consistent with the saturation result:
  more data from the same effective policy doesn't help.

- **Prefer small, high-value self-play additions.** Only add self-play games that
  are qualitatively different from heuristic play (e.g., games where the model
  discovered a novel strategy, or games filtered for unusual positions).

- **Consider epoch ratio, not row ratio.** If mixing 1.28M heuristic + 200k
  self-play, train for 95 epochs on both (total ~140M samples) rather than
  trying to balance row counts by generating more self-play.

### 5.3 When self-play data becomes valuable

Self-play data will become genuinely valuable when the self-play policy is
**meaningfully different** from the heuristic. This requires either:

1. **RL training** (not just BC) that discovers strategies the heuristic never
   plays -- e.g., a policy gradient step that increases probability of a winning
   line the heuristic didn't consider.

2. **MCTS-guided play** where search discovers better moves than the raw policy,
   creating training targets that represent a stronger policy than the one that
   generated them.

3. **Iterative self-play with significant generational improvement** -- but the
   Phase 1 results show this doesn't happen with pure BC. The model just
   reproduces the heuristic.

Until one of these conditions is met, generating more self-play data is
essentially generating more heuristic data at higher compute cost.

### 5.4 Recommended self-play pipeline

1. **Initialize from v99_saturation_1x_95ep** (29.5% combined WR)
2. **Generate self-play data with MCTS** (not raw policy rollouts) to create
   genuinely novel training signal
3. **Keep self-play volume small** (~1-2k games initially) and add to existing
   1.28M heuristic rows
4. **Train for high epoch count** (80-100 epochs) on the combined data
5. **Evaluate** whether the self-play games actually contributed (ablation: train
   with and without the self-play games)
6. Only scale self-play volume if the ablation shows a clear benefit

## 6. Implications for Architecture Evaluation

### 6.1 The variance problem

control_feat h256 shows 4.1pp seed variance (24.8% to 28.9%) compared to
baseline's 3.3pp. This means:

- A single-seed evaluation of control_feat is unreliable (+/-2pp)
- The arch sweep's 29.0% for control_feat was likely a lucky seed, not a
  systematic advantage
- The saturation_1x baseline at 29.5% (single seed) could also be partly lucky

### 6.2 Recommended evaluation protocol

To reliably compare architectures at the current scale:

1. **Minimum 3 seeds per configuration** (42, 7, 123)
2. **Report mean +/- half-range**, not the best seed
3. **Use the 1x @ 95ep protocol** as the standard training recipe (it's our
   strongest single-run result and creates a common compute budget)
4. **Threshold for "real" improvement**: mean difference > 3pp across 3+ seeds
   (roughly 1.5x the half-range of seed variance)

### 6.3 Priority test: control_feat @ 1x95ep

The most informative next experiment is control_feat with the 1x @ 95ep recipe.
If the saturation benefit compounds with control_feat's region scoring features,
we might see 30-32%. If not (or if it shows even higher variance), baseline is
the clear winner.

Run: control_feat h256, nash_b only, 95 epochs, seeds 42 + 7 + 123.

## 7. What This Means for the v99 BC Model as Starting Point

### 7.1 v99_saturation_1x_95ep is the best BC model

At 29.5% combined (46.2% USSR, 13.0% US), this is the strongest behavioral
cloning model we've produced, surpassing:

- All Phase 1 self-play variants (best was v90b @ 27.1%, itself a lucky seed)
- All architecture experiments (control_feat 29.0% was a lucky seed)
- The contaminated-data baseline (combined_bid2_h256 @ 23.1%)

### 7.2 Implications for the training compute budget

The result suggests we were **under-training** in all previous experiments:

- Phase 1 models used 60 epochs on ~2M rows
- The saturation experiment used 95 epochs on 1.28M rows
- The improvement likely comes from *both* cleaner data and more epochs

This means the "plateau" identified in Phase 1 (all models at 23-25%) was
partly an artifact of insufficient training duration on contaminated data.

### 7.3 Upper bound estimate

If the saturation benefit continues to scale:

- 1x @ 150ep: Likely 30-31% (diminishing returns but still some headroom)
- 1x @ 95ep + MCTS data: Potentially 32-35% if MCTS creates novel signal
- 1x @ 95ep + architecture: Potentially 30-32% if control_feat compounds

The practical ceiling for pure BC (no RL, no search) is probably around 32-35%
combined WR against the heuristic. Beyond that requires either RL training
signal or inference-time search.

## 9. Theoretical Framework: Epoch-Data Tradeoff Curve

For a fixed compute budget C = epochs * train_rows:

```
WR(unique_games, epochs) = f(policy_diversity) * g(epochs_per_game) * h(state_coverage)
```

Where:
- `f(policy_diversity)` is sublinear in unique_games when all games come from the
  same policy (log or sqrt scaling)
- `g(epochs_per_game)` is concave but with a long tail -- each epoch extracts
  diminishing but positive signal
- `h(state_coverage)` saturates once the model has seen enough board positions
  to generalize (probably ~5-10k games for h256)

At our current scale (10k games, h256 model):
- `h(state_coverage)` is near saturation for 10k games
- `f(policy_diversity)` is near-flat because nash_b and nash_c are the same policy
- `g(epochs_per_game)` still has headroom at 95 epochs

Therefore: more epochs > more games. This reverses when:
- Data comes from diverse policies (different f scaling)
- Model capacity grows (different h saturation point)
- Training runs much longer (g eventually plateaus too)
